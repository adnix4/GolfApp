1. Phased Delivery Plan — Overview
Golf Fundraiser Pro is delivered in four sequential phases. Each phase produces a complete, usable product that can run a real tournament — later phases add capability rather than replace what was built. Nothing built in Phase 1 is thrown away; each phase activates features that were always planned in the architecture.


Core Principle:  Database schema, API structure, authentication, and organization branding are all designed for the full feature set from day one. Later phases flip switches — they don't require rework.

2. Phase 1 — Admin-Run Tournament
PHASE 1 GOAL:  A complete tournament management system that an organizer can use at a real event today. Golfers register online and check in via QR. The organizer enters scores manually from a tablet. No mobile scoring app, no public leaderboard.


Everything an organizer needs to replace a spreadsheet and paper signup sheet with a real, branded, automated system. This phase could run the CLHS 4th Annual Golf & Give as soon as it's complete.


2.1  Phase 1 Infrastructure (Sprints 1–2, 2 weeks)
* Monorepo setup: Turborepo with apps/admin (Expo Router web), apps/web (Next.js), apps/api (ASP.NET), packages/ui, packages/shared-types, packages/theme
* Docker Compose: PostgreSQL 16 + PostGIS, Redis, pgAdmin, ASP.NET API
* GitHub Actions CI: lint, build, and test pipelines for all apps
* EAS account setup and Vercel account — deploy targets ready even though mobile app is Phase 2
* Database migrations: full schema for all phases (schema is forward-designed; Phase 2+ features just start using the tables)
Why build the full schema in Phase 1:  Building the complete database schema upfront means Phase 2 doesn't require migrations that might break live data. Every table exists from day one — Phase 2 simply starts writing to the ones Phase 1 left empty.


2.2  Phase 1 Feature Deliverables (Sprints 3–10, 8 weeks)
Organization & Event Setup
* Organization creation: name, slug, logo upload, 5-token color theme (Eco Green default), mission statement, 501(c)3 flag
* Event creation: name, event code, format (scramble/stroke/stableford/best-ball), start type (shotgun/tee times), holes (9/18), team size, entry fee model
* Shotgun start: organizer assigns teams to starting holes; system validates no duplicates
* Tee time scheduler: interval-based auto-assignment with manual override; exportable PDF
* Course setup: par per hole, handicap index, yardage — cup GPS coordinates NOT required until Phase 4
* Sponsor configuration: upload logos, assign tiers (Title/Gold/Hole/Silver/Bronze), assign placements


Three-Mode Registration System
* Mode 1 (Full Team): captain registers team; optional invite links for teammates to self-register
* Mode 2 (Join a Team): individual finds team by name or invite token; fuzzy name matching
* Mode 3 (Free Agent): solo player enters pool with preferences (skill level, age, pairing notes)
* Per-event toggle: organizer enables/disables each mode independently
* Partial team support: configurable whether partial teams may submit
* Entry fee payment tracking: paid / pending / waived per team (manual in Phase 1; Stripe in Phase 5+)


Admin Dashboard (Expo Router Web)
* Registration panel: complete teams, partial teams, free agent pool, pending invites — all in one view
* Free agent pairing: drag-and-drop cards into teams; auto-pair button (handicap snake draft)
* Score entry: organizer manually enters team scores hole-by-hole from tablet or laptop
* Hole challenge results: manual entry (closest to pin, longest drive distances, etc.)
* Fundraising panel: entry fee totals, sponsorship amounts, donation log, live grand total
* Conflict resolution: flag and resolve any score discrepancies before publishing results


QR Check-In
* Admin check-in screen: scan player QR, search by name, or browse team list
* Individual and team check-in modes; partial team check-in supported
* Live counter: 'X of 25 teams checked in' with no-show tracking
* Walk-up registration: abbreviated form from check-in screen; QR displayed on-screen for player to screenshot


Admin-Only Leaderboard
* Calculated standings visible to organizer only in the admin dashboard
* Sortable by: to-par, total strokes, Stableford points, holes completed (thru)
* Hole challenge leaderboard: manual results displayed alongside main leaderboard
* NOT publicly accessible in Phase 1 — public leaderboard launches in Phase 2


Organization Landing Page (Next.js)
* Branded event page at /e/{eventCode}: logo, colors, mission statement
* Registration CTA (all three modes), donation widget with thermometer, sponsor showcase
* Countdown timer, remaining team slots, event details block
* SEO: Open Graph tags, JSON-LD Event structured data for Google Search
* 'Live scores coming soon' placeholder where leaderboard link will appear in Phase 2


Automated Emails
* Registration confirmation: event details, team info, QR check-in code
* Team assignment email: sent when free agent is paired (Mode 3)
* Invite link email: sent when captain requests teammate self-registration links
* Event reminder: 24 hours before, with tee time / starting hole and course address
* Thank-you email: final score, leaderboard position, total funds raised, sponsor acknowledgments
* Donation receipt: amount, event name, 501(c)3 note where applicable
* All emails: CLHS/org logo + color theme applied; unsubscribe link included


QR Print Kit & Paper Backup
* QR Print Kit PDF: event join QR, per-player check-in QRs, leaderboard QR (placeholder), donation QR — all in one printable PDF
* Paper scorecard PDF: one per team, pre-filled with hole, par, handicap index — printed at registration desk as backup

3. Phase 2 — Golfer Scoring App + QR Transfer
PHASE 2 GOAL:  Golfers get their own mobile experience. Score entry moves from the admin tablet to each team's phone. The public leaderboard goes live. QR scorecard transfer gives zero-internet score collection at round end. Phase 1 admin capabilities remain and are enhanced.


Phase 2 is the largest technical phase because it introduces the React Native mobile app, the offline SQLite architecture, and the public Next.js leaderboard. However, the API from Phase 1 already handles almost all required operations — the mobile app is primarily a new client on top of the existing backend.


3.1  Mobile App — Core (Sprints 1–4)
Event Join & Team Card
* Golfer opens app → scans event join QR → assigned team card appears
* Team card shows: team name, player roster, starting hole or tee time, event format
* Event config (course pars, hole list, cup coordinates placeholder) cached to SQLite at join time
* App join works offline if the team cache is pre-populated by organizer export


Score Entry UI
* Hole-by-hole score entry: gross score via large ± buttons (56pt minimum touch target for golf gloves)
* Optional per-hole fields: putts, fairway hit, green in regulation
* Player shot contribution tracking: scorekeeper taps which player's drive was selected each hole
* Running cumulative to-par score displayed at top of screen at all times
* Previous hole editable via swipe-back gesture
* Sponsor name shown on score entry screen for hole-sponsored holes


Offline-First Architecture
* Every score tap writes to SQLite (pending_scores table) instantly — before any network call
* App retries sync every 30 seconds in the background via expo-background-task
* Connectivity tier detection: Full (LTE/5G) / Degraded (3G) / Offline / Intermittent
* Offline Event Mode: organizer-configurable flag that increases retry interval and disables live leaderboard on mobile to conserve battery
* Pre-flight checklist: app blocks scoring start until event config is cached


3.2  QR Scorecard Transfer (Round-End Score Collection)
When internet sync is unavailable or incomplete at round end, QR Scorecard Transfer gives the admin a direct, zero-connectivity method to collect scores from each team's phone.


Scorekeeper Side
* Scorekeeper taps 'Generate Scorecard QR' on round complete screen
* App compresses full scorecard to JSON, signs with HMAC-SHA256, encodes as QR
* Screen brightness auto-maximized; QR remains until admin confirms receipt
* Payload > 1,269 bytes: app auto-splits into Part 1 of 2 / Part 2 of 2


Admin Side
* Admin opens 'Collect Scores' on admin device → camera activates
* Camera stays open continuously; chime on each successful scan
* Running count: 'Collected: 18 of 25 teams'
* Scanned scores write to SQLite, flagged as QR-collected, sync to server when connected
* HMAC signature validated before accepting — rejects tampered payloads


QR Payload
* Fields: schema version, event code, team ID (8 chars), team name, device ID, timestamp, HMAC signature, 18-hole score array
* Standard 18-hole payload: ~900 bytes — fits single QR at error correction level M
* Admin collects 25 teams in approximately 8–12 minutes at the 18th green


3.3  Public Web Leaderboard (Next.js)
* Route: /e/{eventCode} — publicly accessible, no login required
* SSR initial render for fast cold load; subsequent updates via polling (Phase 3 upgrades this to SignalR push)
* TV mode at /e/{eventCode}/tv — large-font, auto-scrolling, sponsor banner rotation
* Hole challenge tab — shows manual results entered by admin
* Fundraising thermometer — live donation total
* Sponsor banner — rotating Title/Gold sponsors in header area
* 'Watch Live' link replaces the Phase 1 placeholder on the landing page

3.5  Phase 2 — Low-Connectivity Operations
Phase 2 introduces the full offline capability detailed in Section 7 of this spec. Key additions:

4. Phase 3 — Live Real-Time Scoring + Communications
PHASE 3 GOAL:  Scores push to the leaderboard live during the round — not just at round end. The email advertisement builder ships. Hole challenge leaderboards update in real time. This phase is shorter because Phase 2 already built the infrastructure; Phase 3 flips it on.


Phase 3 is 4–6 weeks rather than 8–10 because all the heavy lifting is done. SignalR is already in the API from Phase 1; it just wasn't connected to score events. Redis is already running. The leaderboard already exists — it just starts receiving push updates instead of polling.


4.1  Live SignalR Score Pushes
* TournamentHub: existing SignalR hub receives ScoreUpdated events when PATCH /scores/{holeId} is called by the mobile app
* All connected leaderboard clients receive the push and re-sort standings client-side
* Admin dashboard also subscribes — organizer sees scores update live without refreshing
* Latency target: < 2 seconds from score submission to leaderboard re-sort
* Graceful fallback: clients without WebSocket support fall back to 15-second SSE polling
* Redis leaderboard cache: 2–5 second TTL absorbs shotgun-burst load spikes


Why Phase 3 not Phase 2:  QR scorecard transfer (Phase 2) gives you a working score collection system that functions perfectly without real-time infrastructure. Live SignalR is a UX enhancement, not a requirement for collecting scores. Separating them means Phase 2 ships faster and with less risk.


4.2  Email Advertisement Builder
* Drag-and-drop builder in Expo Router web admin — pre-populated with event data, sponsor logos, CLHS colors
* QR code linking to registration page embedded automatically (PNG hosted on CDN)
* 11-section email layout: header band, title sponsor banner, event details, mission statement, CTA button, QR block, hole challenge highlights, sponsor rows, donation CTA, footer
* Export formats: HTML file (Mailchimp/Constant Contact ready), ZIP archive, plain text, SendGrid direct send
* PDF preview via headless Chromium (Puppeteer) server-side
* All styles inline for Outlook compatibility; VML button fallback; HTML table layout
* Live preview: desktop and mobile viewport toggleable in the builder


4.3  Hole Challenge Live Leaderboards
* Challenge results update in real time via SignalR ChallengeUpdated event
* Dedicated challenges tab on the public leaderboard: one live leaderboard per active challenge
* Closest to Pin: GPS-based (Phase 4) or manual distance entry; admin can award from dashboard
* Longest Drive: GPS-based (Phase 4) or manual; separate men's/women's categories optional
* Hole-in-One auto-detection: when gross score = 1 on any hole, ChallengeAlert event broadcast to all connected devices
* Hole-in-One celebratory animation displayed on mobile app and leaderboard simultaneously


4.4  Additional Phase 3 Deliverables
* Tee time schedule PDF export from admin dashboard (formatted, printable)
* Score conflict resolution UI polish — improved admin review workflow
* Push notification infrastructure (Expo Notifications) — used for Hole-in-One alert in Phase 3; extended in Phase 4 for GPS approach notifications
* Leaderboard share card generation — OG image with current standings for social sharing


5. Phase 4 — Payments + Auction
PHASE 4 GOAL:  Golfers and attendees pay online, bid on silent auction items from their phones, and participate in a live auction run by an admin host. Donation items let multiple people pledge at any amount. All auction proceeds flow into the real-time fundraising total on the leaderboard.


Phase 4 introduces Stripe payments and the full three-format auction system. Stripe is integrated once and reused across entry fees, auction winner charges, and donation pledges. The auction bidding system uses the SignalR hub already running from Phase 3 — BidPlaced and AuctionExtended events are new message types on the existing connection.


4.1  Stripe Payment Integration
Card-on-File — Stripe Setup Intent
* Stripe Setup Intent saves a payment method without charging — required before any bidding
* Pre-event: golfer is prompted to add a card at the end of the registration flow — 'Save card to bid in the auction and pay entry fees online'
* Day-of: organizer shares a short URL or QR that opens a Stripe-hosted card entry page on the attendee's phone at check-in
* Card stored as Stripe Customer + PaymentMethod; only a reference token saved in the database — no raw card data ever stored
* Bidding eligibility rule: bid endpoint validates has_payment_method = true OR check_in_status = checked_in before accepting any bid


Entry Fee Online Payment
* Stripe PaymentIntent added to registration flow after team creation
* Supports per-player and flat team fee models; webhook payment.succeeded sets entry_fees.paid_at
* Partial tracking: admin dashboard shows 'Fees: 3 of 4 paid' for incomplete teams
* Waiver override: admin marks any fee as waived without requiring payment


Auction Winner Charging
* Silent auction close: system automatically charges the winning bidder's saved card via Stripe PaymentIntent
* Donation pledges: all pledgers charged simultaneously at item close
* Failed charges: admin dashboard alert with manual resolution workflow (re-charge or waive)
* All charges generate SendGrid receipts with item description, amount, FMV for 501(c)3 deductibility, and event name


4.2  Three Auction Formats
Format
	Type Value
	How It Works
	Silent Auction
	silent
	Bidders compete; highest single bid wins. Closes at closes_at with 30-second anti-snipe extension. One winner charged automatically.
	Live Auction
	live
	No automated bidding. Admin host runs room verbally, then enters final amount and selects winner from attendee list. Winner charged automatically.
	Donation Item — Silent
	donation_silent
	Multiple winners — every bidder pledges and is charged. Fixed denominations or open amount. Self-serve on phone all event.
	Donation Item — Live
	donation_live
	Admin assigns each pledge amount and pledger via host screen. All pledgers charged at close. Good for paddle-raise moments.
	

4.3  Silent Auction — 30-Second Extension Rule
How Anti-Snipe Extension Works
* When a valid bid is submitted: if NOW() > closes_at − 30 seconds, then closes_at = MAX(closes_at, NOW() + 30 seconds)
* Extension cap: closes_at may never exceed original_closes_at + max_extension_minutes (default 10 minutes total)
* Beyond the cap no further extensions are granted — auction closes hard at the cap
* SignalR AuctionExtended event pushes the new closes_at to all connected clients immediately — countdown timer resets on every screen
* Admin dashboard shows extension history: 'Extended 3 times — closes 8:34 PM (originally 8:30 PM)'


Bid Validation Rules
* Minimum amount: current_high_bid_cents + bid_increment_cents
* Buy-now: if amount >= buy_now_price_cents, item closes immediately after bid is accepted
* Race condition: bid submission uses a PostgreSQL row-level lock on the auction_items row — two simultaneous bids cannot both win
* Outbid notification: previous high bidder receives Expo push notification + email
* Eligibility: has_payment_method = true OR check_in_status = checked_in


4.4  Live Auction — Admin Host Control Screen
Host Screen (Expo Router Web Admin)
* Admin taps 'Start Live Auction' — activates 'Live Auction in Progress' banner on all public and mobile screens
* Current item shown with large photo, title, description, starting bid, and donor name
* Amount field: admin types the current called amount as the room bids verbally — displayed live on attendee screens
* Winner field: dropdown of all checked-in attendees; admin selects winner and taps 'Award Item'
* Donation item during live: admin enters each pledge amount and pledger — multiple rows, 'Add Pledge' button
* 'Next Item' button: advances queue; pushes LiveItemAdvanced SignalR event to all clients


Attendee Experience During Live Auction
* Mobile app shows 'Live Auction in Progress' tab with current item details and live called amount
* 'I'm Bidding' button: soft paddle-raise indicator — not a formal bid, but visible as a count on the admin host screen
* Donation items during live: attendees tap pledge denomination directly on phone; admin sees pledgers accumulate in real time


4.5  Donation Items — Fund-a-Need / Paddle Raise
Fixed Denomination Mode
* Admin sets preset pledge amounts: e.g. $25, $50, $100, $250, $500 stored as donation_denominations[] (cents)
* Bidding screen shows a button per denomination — attendee taps the amount they want to pledge
* Multiple attendees can pledge the same amount — all charged at close
* Running pledge total shown in real time: '$3,250 pledged — 18 donors'
* Optional goal thermometer: admin sets a target amount; thermometer shown on item detail and public web page


Open Amount Mode
* donation_denominations = null — attendee enters any amount >= minimum_bid_cents
* Useful for campaigns where maximum flexibility is preferred


Both Donation Subtypes Close the Same Way
* closes_at triggers; all pledgers charged simultaneously via Stripe batch PaymentIntents
* Failed charge: admin notified individually; manual resolution
* Receipts include amount, FMV = $0 (full donation), 501(c)3 note, tax year

6. Phase 5 — League Play + Handicaps
PHASE 5 GOAL:  Golf clubs, courses, and membership organizations can run full season leagues: weekly scheduled rounds, automated handicap calculation after every round, Stableford and stroke play standings, flight divisions, balanced weekly pairings, skins games, and a player-facing season dashboard showing handicap trend and standing history.

5.3  Handicap Calculation Engine
Handicap is the most complex new requirement. The engine runs automatically after every round is finalized and updates every member's handicap_index, writing a full audit trail to handicap_history.


Phase 5a: Club/League Formula (Launch)
* Admin configures the formula at league creation — no external API required
* Formula options: Best N of last M rounds (e.g. best 5 of last 10), straight average of last N, or percentage of scoring average (e.g. 85% of last 10 rounds average)
* Each differential = Adjusted Gross Score − Course Par (simplified club formula)
* Or full differential = (Adjusted Gross − Course Rating) × 113 / Slope Rating when course rating and slope are provided
* After every round closes: engine queries each player's last M differentials, applies the formula, writes new index to league_members.handicap_index and handicap_history
* Handicap cap: configurable max index per league (e.g. 36.0) — common for competitive leagues
* Admin manual override: admin can override any player's handicap with a reason field; logged in handicap_history with admin_override = true
* Sandbagger flag: if a player's handicap has not decreased after 5+ rounds significantly below their index, admin dashboard shows a flag for review


Phase 5b: USGA Formula (Enhancement)
* Uses the official USGA Handicap Index formula: best 8 of last 20 Score Differentials
* Each Score Differential = (Adjusted Gross Score − Course Rating) × 113 / Slope Rating
* Requires course rating and slope rating stored per course and per tee set in course_holes metadata
* No GHIN API required for the calculation — GHIN is only needed to sync indexes with the USGA system
* Start USGA GHIN API application during Phase 4; integrate in Phase 5b if approved


Net Score and Course Handicap
* Course Handicap = Handicap Index × (Slope Rating / 113) + (Course Rating − Par) — rounds to nearest integer
* Stroke play net: Gross Score − Course Handicap
* Stableford net: points awarded per hole based on net score vs. par (double bogey = 0, bogey = 1, par = 2, birdie = 3, eagle = 4)
* Match play: handicap strokes allocated per hole by difficulty rank (stroke index 1–18); player receives a stroke on holes numbered ≤ their handicap difference

5.5  Weekly Pairing Engine
The pairing engine runs when an admin opens a new round and taps 'Generate Pairings.' It produces foursomes (or threesomes/twosomes at configurable thresholds) respecting flight boundaries and minimizing repeat matchups.


Algorithm Constraints
* Hard constraint: players in different flights are not matched against each other for standing points (though they may share a tee time physically)
* Hard constraint: absent players (marked in advance or day-of) are excluded from pairings
* Soft constraint: minimize repeat pairings — the engine tracks pairing history and penalizes repeat matchups in the current season
* Soft constraint: for match play, balance wins/losses so no player plays the same opponent more than once until everyone has played everyone
* Substitute handling: admin adds a substitute player with a temporary handicap; sub's round is counted for the substitute's record, not the absent member's


Output
* Pairings displayed as a tee sheet: group number, players, tee time, starting hole
* Pairings PDF generated and available to print at the first tee
* Players notified via push notification and email: 'Round 7 pairings are ready — you tee off at 8:42 AM in Group 4 with John, Mike, and Sarah'
* Admin can manually drag-and-drop players between groups before locking pairings

* Round Management Create round → generate pairings → open scoring → close round → publish results → trigger handicap update. Each step is a distinct admin action.
* Handicap Manager View all current handicaps, full history per player, manual override with reason, sandbagger flags, handicap trend chart

7. Phase 6 — GPS & Mapping
PHASE 6 GOAL:  Golfers see their distance to the pin in real time, with elevation change, on a Mapbox satellite view of the hole. Drive distance tracking enables GPS-based longest drive contests. Kept as its own phase because it has the highest technical risk and benefits from a stable platform underneath it.


Phase 6 introduces genuinely different technical requirements: PostGIS spatial queries, Mapbox GL integration, DEM terrain elevation API, and on-course device GPS behavior. The database schema has had the cup_location column ready since Phase 1 — Phase 6 fills it in. GPS is also highly valuable for league play, where players return to the same course every week and will quickly memorize exact yardages.


6.1  Cup GPS Location Database
* course_holes.cup_location: geometry(PointZ, 4326) — exists in schema from Phase 1, first populated in Phase 6
* course_holes.cup_elevation_m: elevation in meters from DEM API, stored at cup setup time
* Admin 'Course Setup' tool in mobile app: organizer walks each hole, taps 'Set Pin Location'
* One-time setup per course; reusable across events and league rounds; admin updates pin position on event day
* PostGIS GIST spatial index on cup_location for fast proximity queries


6.2  Distance to Pin — Offline-Capable
Distance to pin works with zero connectivity once cup coordinates are cached at event join. This is the most resilient GPS feature — and particularly valuable for league courses where tiles are cached once and reused weekly.


* Cup coordinates downloaded to SQLite at event/round join — < 5 KB for all 18 holes
* Online mode: GET /holes/{id}/distance?lat=&lng= — server calculates via PostGIS ST_Distance
* Offline mode: Haversine formula runs client-side in JavaScript — < 1ms, no network required
* Display: 'XX yds to pin' with elevation: '+12 ft uphill' or '−8 ft downhill'
* Elevation: DEM-keyed terrain elevation — far more accurate than device GPS altitude (±10–30 ft)
* GPS polling: every 15 seconds when hole map active; paused on score entry screen to conserve battery
* Uncertainty ring: ± confidence indicator shown for wooded or covered holes


6.3  Mapbox Satellite Hole View
* Mapbox satellite tile layer as base map; cup marker, player dot, distance line with yardage label
* Map auto-rotates to face direction of play (GPS course-of-travel heading, not compass)
* Yardage rings: 100, 150, 200 yard concentric circles togglable by golfer
* Toggle: satellite view vs. hole layout graphic
* Offline tile cache: ~150–250 MB per 18-hole course — downloaded before round, cached for weekly reuse in league mode


GPS Accuracy Note:  Device GPS accuracy is 3–5 meters open, 10–30 meters under canopy. Display distances as approximate. Mark wooded holes in course config to disable GPS drive distance. Use DEM terrain data, not device altitude.


6.4  GPS Drive Distance Tracking
* 'Mark Drive' and 'Mark Ball' buttons in score entry; GPS coordinates stored in scores table
* Drive distance displayed with ± disclaimer; manual entry always available
* Longest drive contest: longest GPS drive per hole surfaced on challenge leaderboard
* Wooded holes: GPS drive disabled per hole config; manual entry required


6.5  Hole Layout Graphics
* Organizer uploads hole layout PNG/SVG per hole in admin Course Setup tool
* Overlay on Mapbox satellite view with alignment handles
* Toggle between satellite and hole graphic
* Auto-generated schematic from GPS boundary polygon is Phase 7+ feature

9. Architecture Decision — Expo Router Web vs. Next.js
Evaluated with Expo SDK 53–55 and the New Architecture now default. The question: can the complete UI be built in React Native / Expo, abandoning Next.js?


DECISION: Hybrid Architecture  Admin Dashboard moves to Expo Router web. Public leaderboard, landing page, and email ad builder remain in Next.js. Full abandonment of Next.js is not recommended at this time.


9.1  Why Admin Dashboard → Expo Router Web
* No SEO requirement — admin is behind authentication, never indexed by search engines
* Shared components: score cards, team lists, check-in UI built once in packages/ui and rendered on both mobile and web admin
* EAS Hosting handles deployment — one fewer platform to manage
* Admin auth uses httpOnly cookies on web — supported by Expo Router middleware


9.2  Why Leaderboard & Landing Page Stay in Next.js
* Expo Router SSR is alpha in SDK 55 — not appropriate for public SEO-critical surfaces
* Landing page needs ISR for donation counter freshness — not available in Expo Router
* TV mode requires CSS media queries, VH/VW units, scroll-snap — no clean React Native Web equivalent
* JSON-LD Event structured data (Google Search rich results) is simpler in Next.js generateMetadata()
* Email advertisement HTML export (DOM + Puppeteer) is inherently web-native
