1. How to Use This Document Series
Golf Fundraiser Pro is built in six phases. Each phase has its own implementation design document. This Foundation document covers everything that is shared across all phases — the monorepo structure, the complete database schema, the authentication system, the theme engine, and the DevOps pipeline. Read this document once before starting any phase. Do not duplicate its content in phase-specific documents; reference it instead.


Token Efficiency:  When working with an AI assistant on a specific phase, load this Foundation document plus that phase's document. You do not need to load all six phase documents simultaneously. The Foundation + one Phase document is sufficient context for implementing that phase.

4. Complete Database Schema
The full schema is created in Phase 1 migrations. All tables exist from day one — later phases write to columns that were always there. This prevents breaking migrations on live data. Columns populated in later phases are nullable until that phase is deployed.


Migration Strategy:  Use EF Core migrations with a single DbContext. Each phase adds a named migration: Phase1_InitialSchema, Phase2_AddOfflineColumns, Phase4_AddAuction, Phase5_AddLeague, Phase6_AddGPS. Never drop or rename existing columns — add new ones.

5. Authentication & Authorization
5.1  JWT Flow
All protected API endpoints require a Bearer JWT. The access token is short-lived (15 minutes). The refresh token is long-lived (30 days, rotating).
* POST /auth/register — create organizer account (email + password). Returns access + refresh tokens.
* POST /auth/login — validate credentials. Returns access + refresh tokens.
* POST /auth/refresh — exchange refresh token for new access + refresh pair. Old refresh token invalidated immediately.
* POST /auth/logout — invalidate refresh token server-side.

5.3  Role System
* OrgAdmin — full access to their organization's events, members, and settings
* EventStaff — check-in and score entry for a specific event; no settings access
* Golfer — score entry for their own team; read leaderboard; bid in auction
* Public — read-only leaderboard and event landing page; no auth required
* Role is stored in the JWT claims. API endpoints use [Authorize(Roles = "OrgAdmin")] attributes.

6.2  Token Distribution by Platform
* Next.js: CSS custom properties injected in root layout.tsx at SSR time — zero flash of unstyled content.
* Expo Router web: same CSS custom properties via shared theme package.
* React Native mobile: ThemeContext provides tokens via useTheme() hook to all components.
* Email: inline style substitution at SendGrid render time — {{PRIMARY_COLOR}} replaced with org hex.
* Per-event override: events.config.theme_override JSONB (nullable) — null inherits org theme.


7. Shared Packages
7.1  packages/shared-types
TypeScript interfaces and Zod schemas for every API request and response. Imported by both the API (for request validation) and all frontend apps (for type-safe data fetching). Never define the same DTO shape in two places.
* EventDTO, TeamDTO, PlayerDTO, ScoreDTO, LeaderboardDTO
* AuctionItemDTO, BidDTO, AuctionWinnerDTO
* LeagueDTO, SeasonDTO, LeagueMemberDTO, StandingsDTO
* All enums re-exported from here so they're shared across API and frontend


7.2  packages/ui
React Native components shared between the mobile app (apps/mobile) and the admin dashboard (apps/admin). Components must work in both React Native and React Native Web (Expo Router web output).
* ScoreCard — per-hole gross score entry with ± buttons and 56pt touch targets
* LeaderboardRow — team name, to-par, thru, sponsor badge
* TeamCard — player roster, starting hole, tee time, check-in status
* SponsorBanner — logo + tier badge, used in scoring screens and leaderboard
* ThemeProvider — wraps app root; provides useTheme() hook
* Rule: no platform-specific code (no Platform.OS checks) in packages/ui. Use StyleSheet.create for all styles.


7.3  packages/theme
* ECO_GREEN_DEFAULT — the 5-token default palette as a typed object
* validateContrast(primary, surface) → boolean — WCAG 4.5:1 check using relative luminance formula
* buildCSSVars(theme) → string — generates :root { --color-primary: #... } block for web
* buildThemeContext(theme) → ThemeContextValue — object for React Native ThemeContext

10. API Conventions
All API endpoints follow these conventions. Phase documents define specific endpoints; they all comply with these rules.
* Base URL: /api/v1/...
* Authentication: Authorization: Bearer {accessToken} header on all protected routes
* Request bodies: JSON with Content-Type: application/json
* Pagination: cursor-based where needed. ?after={cursor}&limit={n} (default 50, max 200)
* Timestamps: ISO 8601 UTC strings in all request/response bodies
* Errors: { error: string, code: string, details?: object } with appropriate HTTP status
* IDs: UUID strings (not integers) in all request/response bodies
* Soft deletes: records are never deleted from the database. Use status = cancelled or active = false.
