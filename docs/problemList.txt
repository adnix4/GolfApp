## Security review (2026-06-22)
Findings from the security review, ordered for remediation. Mark [x] when done.

[x] 1. (HIGH) Stripe webhook signature bypass — apps/api/Features/Webhooks/StripeWebhookController.cs:39-52.
       DONE 2026-06-22: webhook now fails closed (500) outside Development when the secret is unset.
       When STRIPE_WEBHOOK_SECRET is unset the [AllowAnonymous] webhook parses the
       event WITHOUT verifying the Stripe-Signature. payment_intent.succeeded flips
       an AuctionWinner to ChargeStatus.Succeeded + ReceiptSent, so an attacker can
       forge that event (with winner_id metadata) and mark auction items paid with
       no money moved. Fix: fail closed outside Development when the secret is unset.

[x] 2. (MEDIUM) No rate limiting on the API — AddRateLimiter/UseRateLimiter appear
       DONE 2026-06-22: added .NET rate limiter — global 600/min per-IP backstop + named policies
       (auth 20/min on login/register/refresh, join 60/min, donate 30/min); UseRateLimiter wired
       after UseRouting; JoinEvent's false comment corrected. (sync covered by global backstop.)
       nowhere. Unthrottled anonymous POSTs: /events/{code}/join (email enumeration),
       /auth/login (brute force; only per-account Identity lockout exists), /auth/register
       (org spam), /sync/scores (score flooding), /pub/events/{code}/donate (fake-donation
       flooding). MobileController.JoinEvent's doc-comment FALSELY claims "Rate-limited by
       event code." Fix: add a global per-IP limiter + stricter named policies on the
       anonymous POSTs (UseRateLimiter after UseRouting); then implement-or-delete the comment.

[x] 3. (MEDIUM) IDOR on player self-update — apps/api/Features/Mobile/MobileService.cs:481
       DONE 2026-06-22: UpdateSelfRequest now requires EventCode + Email; the service verifies them
       against the player record (case-insensitive) before applying the edit and throws the same
       NotFound on mismatch (no existence leak). Mobile client (api.ts updateMyProfile + edit-profile.tsx)
       sends the join email/eventCode as proof.
       FULLY CLOSED 2026-06-22 by the join token (Phase11): UpdateSelf now requires the per-player
       session_token minted at /join (email/eventCode proof removed), closing teammate-to-teammate too.
       Original: PATCH /api/v1/players/{playerId}/self was anonymous, loaded the player by id with
       NO binding to the caller, and playerIds are exposed in join/scorecard/team responses, so
       anyone could edit any player's firstName/lastName/phone.

[x] 4. (LOW/hardening) JWT_SECRET length unchecked — apps/api/Common/Extensions/ServiceCollectionExtensions.cs:172.
       DONE 2026-06-22: startup now throws if JWT_SECRET < 32 bytes; check-updates.js validates the
       package name against /^@?[a-z0-9._/-]+$/i before the shell `npm view` call.
       A short secret weakens HMAC-SHA256. Enforce >= 32 bytes at startup (throw otherwise).
       Also harden scripts/check-updates.js: the `npm view ${name}` call runs through a shell;
       name is registry-sourced (not user input) so risk is low, but switch to execFileSync /
       validate name against /^@?[a-z0-9._/-]+$/i for defense-in-depth.

[x] 5. (LOW/by-design) Anonymous score injection — /api/v1/sync/scores trusts

TOP PICKS (high payoff)
[-] M1. expo-sqlite session changesets -> simplify offline sync (SIMPLIFY CODE).
        SPIKED 2026-06-23: REJECTED — does not fit this architecture.
        Original idea: capture local mutations as a changeset and apply/rebase server
        changesets natively, replacing pending_scores + backgroundSync.ts +
        mergeServerScores + the admin-wins / proposed_score conflict flow with DB
        primitives. Findings:
        - Session changesets are a SQLite<->SQLite BINARY replication format
          (create/applyChangesetAsync exchange an ArrayBuffer of SQLite's changeset
          format; see node_modules/expo-sqlite/build/NativeSession.d.ts). The headline
          benefit "apply server changesets natively" requires the server to be a SQLite
          peer. Ours is ASP.NET Core + PostgreSQL with JSON-REST sync
          (POST /api/v1/sync/scores {scores:[...]} -> {accepted,conflicts}; pull-back
          via TeamScorecard JSON). It cannot emit/consume SQLite changesets, so that
          benefit is unreachable without rebuilding backend sync around SQLite — a
          non-starter for a Postgres backend.
        - store.mergeServerScores is domain-specific bookkeeping: protect unsynced
          local edits (synced_at IS NULL), else adopt server value, set the `conflict`
          flag for the proposed_score approval flow, manage completed_at. A changeset
          conflict handler only does generic row/column ops (OMIT/REPLACE/ABORT) and
          can't express those rules, so the app logic stays either way.
        - Dataset is ~18 rows (1/hole/team), single table, deterministic PK, already a
          clean INSERT OR REPLACE upsert. No large multi-table mutation set for
          changeset replication to optimize.
        Net: adds a native session-extension dependency + binary-blob plumbing while
        removing none of the REST protocol or domain rules. Not worth it.
        Orthogonal note: if offline sync ever needs simplifying, the real smell is
        pending_scores multiplexing 3 roles (editable scorecard / outbound queue /
        server cache) via flag columns — a manual table-split refactor, unrelated to
        changesets.
