# Problem List

## Deployment path — real-world pilot (2026-07-09)
Goal: deploy Golf Fundraiser Pro for a real-world test event. Ordered phases;
mark [x] as completed. D1 items have external lead times — start them first,
they run in parallel with everything else.

[ ] D1. DECISIONS & ACCOUNTS (user actions; longest lead times)
        - Hosting choice. Spec default: API on Railway (any container host
          works), web on Vercel, admin as a static expo export served
          anywhere, managed Postgres+PostGIS and Redis. A single VPS with
          docker-compose + reverse proxy is also fine at pilot scale.
        - Domain + DNS (.env.example assumes golffundraiser.pro with api. and
          app. subdomains).
        - Apple Developer Program enrollment ($99/yr; TestFlight review adds
          days). SDK 56 constraint: shipped iOS builds require iOS 16.4+
          (Xcode 26.4 — EAS cloud runners handle it, Windows dev unaffected,
          but older devices are dropped). Confirm EAS image + minimum-OS
          commitment before releasing a build.
        - Stripe live account activation (business verification lead time) —
          or run the pilot in test mode and use the cash mark-paid-at-check-in
          path for real fees.
        - Email provider (SendGrid or similar) + SPF/DKIM domain auth.
          CRITICAL PATH: join email-verification codes must actually arrive
          once the dev bypass code is off (see A3 history in git).

[ ] D2. CODE HARDENING (no infra needed; can be done now)
        - Startup guard: refuse to boot in Production when
          JoinVerification:TestBypassCode is set (converts the "999999 must
          never reach production" note into a can't).
        - ForwardedHeaders configuration so the rate limiter and the
          IP-keyed security policies (auth/join/donate) see real client IPs
          behind the reverse proxy — the code comments already flag this.
        - DataProtection key persistence to a mounted volume/store so keys
          survive container restarts (currently Windows DPAPI locally).
        - Production CORS origins from configuration.
        - /health endpoint for host health checks (verify one exists).
        - Migration story: auto-migrate is Development-only by design, so
          deploys need an explicit `dotnet ef database update` step or an
          opt-in migrate-on-startup flag for production.
        - DEPLOYMENT.md: complete env-var checklist (extend .env.example's
          STAGING/PROD annotations) + deploy runbook per app.

[ ] D3. STAGING ENVIRONMENT (after D1 hosting/domain decided)
        - Provision managed Postgres WITH PostGIS (plain postgres breaks GPS
          features — see infra/docker-compose.yml warning), Redis, and object
          storage for uploads (IFileStorage S3/R2 support is merged and
          MinIO-verified; local-disk uploads don't survive redeploys).
        - Deploy API + web + admin static export; set NEXT_PUBLIC_* /
          EXPO_PUBLIC_* build-time URLs; register the Stripe webhook endpoint
          (fail-closed outside Development is already in); TLS everywhere.

[ ] D4. DRESS REHEARSAL on staging (the pilot succeeds or fails here)
        - /seed-demo-event full lifecycle against staging.
        - Score from real phones, including airplane-mode offline entry and
          sync-back on reconnect.
        - Verification emails arriving on real inboxes (Gmail/Outlook).
        - Stripe test charge + webhook end-to-end; cash mark-paid path.
        - TV leaderboard mode on an actual screen.

[ ] D5. MOBILE DISTRIBUTION
        - EAS builds: iOS via TestFlight, Android via internal testing track
          (or direct APK for a friendly pilot). Expo Go is NOT viable for a
          real event — golfers need an installable build.
        - Push notification credentials (EAS project) verified end-to-end.

[ ] D6. PILOT OPS
        - Nightly pg_dump backups; error monitoring (e.g. Sentry in all three
          apps); uptime ping on / and /health.
        - One-page day-of runbook: who has admin access, cash-payment
          fallback if Stripe hiccups, admin-entry scoring fallback if a
          phone fails.
        - Pick a small, friendly event; run a test-mode preview with the
          organizer the week before.
